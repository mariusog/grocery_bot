## Code Review

### Bug #1: CRITICAL -- Multi-bot same-position deadlock

**Severity:** Critical (all multi-bot games score 0)

**Location:** `bot.py`, lines 362-370 (anti-collision blocked set construction)

**Description:** When multiple bots spawn at the same position (which the simulator does for all multi-bot configs), the anti-collision logic adds the other bot's position to the `blocked` set. Since both bots share a position, each bot's OWN position is in the blocked set. The `bfs()` function does a reverse search from the goal and checks `if npos in blocked` -- when it tries to reach the bot's start position, it finds it blocked and skips it. Result: BFS never finds a path, returns None, and the bot waits forever.

**Root cause:** Line 370: `blocked = blocked_static | other_bot_positions`. When bot A is at (10,8) and bot B is also at (10,8), `other_bot_positions` for bot A includes (10,8), which is bot A's own position. BFS reverse-searches from goal to start, and start=(10,8) is in blocked, so BFS fails.

**Fix:** Exclude the current bot's own position from the blocked set:
```python
# In the bot loop, after building other_bot_positions:
other_bot_positions.discard(pos)  # Don't block our own position
```

**Impact:** This single fix would enable all multi-bot configurations to function. Current expected improvement: 2-bot scores should roughly double single-bot scores due to parallelism.

### Bug #2: Potential issue -- `_blocked_static` identity check

**Location:** `bot.py`, line 69

**Description:** `get_distances_from()` uses `blocked is _blocked_static` (identity check, not equality). This works correctly in practice because the code always passes the actual `_blocked_static` reference for static lookups. However, if `blocked_static | other_bot_positions` were ever passed, it would create a NEW set object and bypass the cache -- which is actually the intended behavior (dynamic blocked sets should not be cached). But this is fragile and non-obvious.

**Severity:** Low (works correctly today, but could break with refactoring)

### Bug #3: Duplicate step number comment and variable scope

**Location:** `bot.py`, lines 493 and 530

**Description:** Two different code blocks are both labeled "# 3." (step 3). The first is "Opportunistic preview pickup" and the second is "If inventory full, go deliver." The variable `picked_up_preview` set in the first block is only used within that block, but the dual numbering makes the logic harder to follow.

**Severity:** Low (cosmetic / readability)

### Performance Observations

1. **BFS caching is very effective:** Only 39 `bfs_all` calls for a 300-round game. The `_dist_cache` stores distance maps from previously visited positions, so repeated lookups are O(1).

2. **TSP brute-force is fine for current order sizes:** Orders have 3-4 items, so TSP evaluates at most 4! = 24 permutations. This would become problematic only for orders with 8+ items.

3. **`plan_multi_trip` uses combinations + TSP:** For 4-item orders with capacity 3, it evaluates C(4,3) = 4 splits, each with TSP on both halves. Total: ~4 * 2 * 6 = 48 permutations. Very fast.

4. **No unnecessary recomputation in the hot path:** `decide_actions` avoids recalculating static data. The `init_static` call on round 0 is the only expensive operation.

5. **Multi-bot `tsp_route` inflation:** In multi-bot mode, `tsp_route` is called ~10x more because bots endlessly re-plan routes they never execute (collision bug). Fixing bug #1 would also fix this waste.

### Correctness Analysis

1. **Optimal decisions (1-bot):** The bot makes near-optimal decisions for the 1-bot case. TSP routing, multi-trip planning, and preview pipelining all work correctly. Score of 126 on Easy with seed 42 is strong.

2. **Endgame behavior:** The `round_trip < rounds_remaining` check (line 575) correctly skips items that cannot be fetched and delivered in time. The round-trip cost includes distance from bot to adjacent cell + 1 (pickup) + distance from cell to dropoff. This is correct.

3. **Preview cascade priority:** The bot correctly distinguishes between cascade-worthy preview items (types NOT in active order, which auto-deliver on order completion) and same-type preview items (which get consumed by the active order's drop_off). Cascade items get a larger detour budget (6 rounds vs 3).

4. **Fair item distribution (multi-bot):** The `max_claim_per_bot` logic divides items fairly among idle bots. This works in theory but is untestable due to bug #1.

### Improvement Opportunities

1. **Fix multi-bot collision (Bug #1):** Single highest-impact change. Would unlock 2-bot and 5-bot play.

2. **Smarter bot coordination:** After fixing bug #1, bots could benefit from explicit role assignment (e.g., bot 0 handles trip 1, bot 1 handles trip 2) instead of each bot independently claiming items.

3. **Endgame partial delivery optimization:** In the last 30 rounds, the bot could calculate the maximum-scoring subset of items it can deliver rather than greedily chasing the closest ones.

4. **Order look-ahead beyond preview:** Currently only considers the next (preview) order. Looking 2-3 orders ahead could enable better item pipelining.

5. **Dynamic detour thresholds:** The fixed detour budgets (3 rounds for same-type, 6 for cascade) could be made proportional to rounds remaining and items still needed.
